# 审阅修订系统重构设计文档

## 项目概述

**目标:** 将现有的review-revision skill拆分为"主skill + 6个独立检查skill"的架构，实现模块化、可扩展的审阅修订系统。

**核心设计:** 一个主skill控制审阅流程，调度其他检查skill完成审阅，汇总报告后统一处理。

## 整体架构

### 目录结构

```
.opencode/skills/
├── review-revision/          # 主skill - 调度审阅流程
├── check-character/          # 角色一致性检查
├── check-setting/            # 世界观设定检查
├── check-timeline/           # 时间线连贯检查
├── check-plot/               # 情节逻辑检查
├── check-content/            # 违禁词/合规检查
└── check-style/              # 叙事风格检查
```

### 工作流程

1. 用户调用 `/skill review-revision`
2. 主skill加载项目配置和选定章节
3. 主skill依次调度6个检查skill执行检查
4. 各检查skill返回问题列表（位置、类型、严重程度、描述、建议）
5. 主skill汇总所有问题，按严重程度排序生成统一报告
6. 用户确认修改
7. 主skill更新章节状态为reviewed

### 检查skill接口约定

**输入:**
- 章节内容（Markdown格式）
- 项目配置（novel-project.yaml中的world_building、outline等）

**输出:**
- 问题列表，每项包含：
  - `位置`: 段落号或行号
  - `类型`: 检查类型标识
  - `严重程度`: 错误/警告
  - `描述`: 问题描述
  - `建议`: 建议修改方案

### 严重程度分级

- **错误**: 必须修复的矛盾或违规（如人物特征矛盾、世界观规则违反）
- **警告**: 建议改进但不影响基本逻辑（如风格不统一、表达冗余）

## 各检查skill职责

### check-character（角色一致性检查）

**职责:** 检查章节中人物相关设定是否与配置一致

**检查内容:**
- 人物外貌、性格特征是否与设定一致
- 人物行为是否符合其性格和背景
- 人物关系是否与设定一致
- 人物称呼是否统一
- 已死亡/离开的角色是否错误出现

**依赖配置:** world_building.characters

### check-setting（世界观设定检查）

**职责:** 检查章节中世界观设定是否被遵守

**检查内容:**
- 是否违反已定义的世界观规则（魔法、科技等）
- 地点描述是否与设定一致
- 时代背景是否一致（语言、物品、技术等）
- 社会制度、文化习俗是否被遵守

**依赖配置:** world_building.setting

### check-timeline（时间线连贯检查）

**职责:** 检查章节中时间线是否连贯合理

**检查内容:**
- 事件顺序是否符合逻辑
- 时间跨度是否合理（如"三天后"是否真的合理）
- 人物年龄、成长时间是否一致
- 前后章节时间是否衔接

**依赖配置:** outline.chapters, world_building

### check-plot（情节逻辑检查）

**职责:** 检查章节中情节逻辑是否合理

**检查内容:**
- 情节发展是否有合理的因果关系
- 是否存在未解决的情节线
- 人物动机是否充分
- 是否有情节漏洞或矛盾

**依赖配置:** outline.chapters, world_building.characters

### check-content（违禁词/合规检查）

**职责:** 检查章节内容是否合规

**检查内容:**
- 是否包含用户定义的违禁词
- 是否包含敏感内容（暴力、色情等，按用户设定级别）
- 是否符合目标平台的发布规范
- 是否有不当的政治/宗教内容

**依赖配置:** consistency_rules（可扩展违禁词列表）

### check-style（叙事风格检查）

**职责:** 检查章节叙事风格是否一致

**检查内容:**
- 叙事视角是否一致
- 文风、语气是否统一
- 对话风格是否符合角色身份
- 是否有重复句式或冗余表达
- 段落长度、节奏是否合理

**依赖配置:** 无（主要基于文本分析）

## 主skill调度逻辑

### review-revision工作流程

1. **加载项目配置**
   - 读取novel-project.yaml
   - 确认存在drafted状态的章节
   - 完成标准: 成功读取配置

2. **选择要审阅的章节**
   - 列出所有drafted状态的章节
   - 用户选择要审阅的章节
   - 完成标准: 用户选择一个章节

3. **调度检查**
   - 依次调用6个检查skill
   - 收集每个检查返回的问题列表
   - 完成标准: 所有检查执行完成

4. **生成汇总报告**
   - 合并所有问题列表
   - 按严重程度排序（错误在前，警告在后）
   - 标注每个问题的来源检查
   - 完成标准: 报告生成

5. **用户确认修改**
   - 展示统一报告
   - 用户逐项确认修复或忽略
   - 完成标准: 用户确认所有修改

6. **更新状态**
   - 保存修改后的章节
   - 更新chapters.reviewed列表
   - 完成标准: 配置文件成功更新

### 统一报告格式

```markdown
# 审阅报告 - 第X章

## 错误（必须修复）

| # | 来源 | 位置 | 问题 | 建议 |
|---|------|------|------|------|
| 1 | check-character | 第3段 | 张三眼睛颜色与设定不符 | 将"蓝色眼睛"改为"黑色眼睛" |
| 2 | check-timeline | 第7段 | 时间跳跃不合理 | 补充过渡说明 |

## 警告（建议改进）

| # | 来源 | 位置 | 问题 | 建议 |
|---|------|------|------|------|
| 1 | check-style | 第5段 | 对话风格与角色身份不符 | 调整用词更符合角色背景 |
```

## 各检查skill的SKILL.md结构

每个检查skill的SKILL.md包含：

```markdown
---
name: check-xxx
description: xxx检查
---

# xxx检查Skill

## 职责
[一句话说明职责]

## 输入
- 章节内容（Markdown格式）
- 项目配置（novel-project.yaml中的xxx部分）

## 检查规则
[详细检查规则列表]

## 输出格式
问题列表，每项包含：
- 位置: 段落号或行号
- 类型: 检查类型标识
- 严重程度: 错误/警告
- 描述: 问题描述
- 建议: 建议修改方案

## 错误处理
- 配置文件不存在: 提示用户先运行novel-project skill
- 章节内容为空: 提示用户先完成章节撰写
```

## 与现有系统的关系

**替换关系:**
- 新的review-revision + 6个check-skill替换现有的review-revision
- 现有的review-revision SKILL.md将被删除

**保持不变:**
- novel-project.yaml配置结构不变
- 章节文件格式不变
- 其他skill（novel-ideation, world-building, outline-design, chapter-writing, polish-style）不变

## 成功标准

1. 7个skill全部实现，职责清晰
2. 主skill能正确调度所有检查skill
3. 统一报告格式清晰，按严重程度排序
4. 每个检查skill可独立调用
5. 错误处理完善
